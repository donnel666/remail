# ADR-019：内部账本采用六位小数精度

## 状态

Accepted（2026-07-10）

## 背景

项目商品价格已经使用 `DECIMAL(18,6)`，并需要支持 `0.008` 接码售价、`0.005` 接码结算价和 `0.007` 购买结算价。原订单、钱包和流水仅保留两位小数，导致商品展示值与实际扣款值不一致。

## 备选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| 商品价格限制为两位小数 | 改动最小 | 无法表达已确认的业务价格 |
| 只把订单金额改为三位小数 | 能正确创建订单 | 钱包、退款和流水仍会丢失精度 |
| 内部账本统一六位小数 | 与商品价格一致，扣款和退款闭环精确 | 需要数据库迁移和全链路回归 |

## 决策

- 商品报价、订单实付/退款、钱包余额、钱包流水、累计消费和返佣事实统一使用 `DECIMAL(18,6)`。
- 领域/API 金额字符串至少保留两位小数；存在分以下金额时保留至多六位，示例：`10.00`、`0.008`。
- 充值额度和卡密面额属于站内钱包额度，统一使用六位小数；仅支付渠道实际收款金额 `payment_amount` 保留两位小数。
- 工作台和余额类界面最多显示三位小数；数据库和领域层不得按展示精度提前舍入。

## 影响

- `0.008` 的公开库存订单必须实际扣除 `0.008`，退款也必须精确退回 `0.008`。
- 旧的整分金额 API 表示保持兼容，例如 `10.00` 不变。
- 迁移降级到两位小数会损失分以下精度，只用于开发/测试回滚。

## 验证要求

- 报价、扣款、订单落库、钱包余额和退款必须覆盖分以下金额测试。
- 数据库迁移测试必须确认金额列的 `NUMERIC_SCALE=6`。
- 前端必须覆盖 `￥0.008` 和窄布局下的完整金额可访问性。
